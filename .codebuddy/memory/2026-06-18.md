## V28 — 服务器重启后 Coze 连接状态重置为未连接

### 需求
服务器重启时，所有用户的 Coze 连接状态应变为"未连接"，提示用户重新授权。

### 实现方案
利用 `system_config` 表存储 `server_startup_at` 时间戳，每次启动时更新。检查 Coze token 有效性时，比较 `coze_tokens.updated_at` 与 `server_startup_at`：如果 token 更新时间早于服务器启动时间，则标记为"未连接"。

### 修改文件

1. **`src/server.ts`** — 启动时写入 `server_startup_at`：
   - `app.prepare().then()` 回调改为 `async`
   - 启动时执行 `INSERT INTO system_config ... ON DUPLICATE KEY UPDATE` 写入当前时间

2. **`src/lib/coze-token.ts`** — 新增 `isCozeTokenStale()` 函数：
   - 查询 `system_config.server_startup_at`
   - 查询 `coze_tokens.updated_at`
   - 比较：`token_updated < server_startup` → 返回 `true`（token 过期）
   - 异常时返回 `false`（不阻塞正常流程）

3. **`src/app/api/user/status/route.ts`** — 导入 `isCozeTokenStale`，判定增加 `&& !isStale`

4. **`src/app/api/tools/route.ts`** — 同上

5. **`src/app/api/tools/[id]/route.ts`** — 同上（之前不检查过期，现在也加了 stale 检查）

### V28 修复 — `isCozeTokenStale` 兜底 created_at
问题：`saveCozeToken` 的 INSERT 语句没有设置 `updated_at`，导致首次 OAuth 授权的用户 token 的 `updated_at` 为 NULL，`isCozeTokenStale` 返回 false。

修复：
1. INSERT 语句加上 `created_at = NOW(), updated_at = NOW()`
2. `isCozeTokenStale` 中同时查询 `updated_at` 和 `created_at`，优先用 `updated_at`，兜底 `created_at`

### V28 修复2 — 时区不一致导致比较失效
问题：`server.ts` 用 `new Date().toISOString()` 生成 UTC 时间写入 `server_startup_at`，而 `coze_tokens.created_at` 用 MySQL `NOW()` 写入（北京时间），导致比较时 `10:56:54` > `02:58:56`（UTC），`isStale` 始终为 false。

修复：`server.ts` 中 `server_startup_at` 改用 `NOW()` 写入，与 token 时间时区一致：
```sql
INSERT INTO system_config (`key`, value, updated_at) VALUES ('server_startup_at', NOW(), NOW())
ON DUPLICATE KEY UPDATE value = NOW(), updated_at = NOW()
```

## V30 — 小程序工具详情页不显示提示词（prompt_info）

### 问题
用户进入工具详情页时，有时不显示提示词（prompt_info 为空），去后台重新连接 Coze 后再回来就显示了。

### 根因分析
1. **后端**：`GET /api/tools/[id]` 的 prompt_info 是从 Coze 官方 API 实时拉取的（`/v1/bots/{coze_id}`），需要有效 Coze Token。如果服务器重启导致 `isCozeTokenStale` 返回 true（token 更新时间早于 `server_startup_at`），`cozeConnected` 为 false，Bot 信息拉取失败，prompt_info 返回 null。

2. **小程序端**：`tool-detail.js` 只有 `onLoad()` 加载数据，**没有 `onShow()` 刷新**。用户从其他页面（如后台）重新连接 Coze 后返回详情页，页面数据仍是旧的（prompt_info 为 null）。

### 修复
在 `tool-detail.js` 中：
- **添加 `onShow()` 生命周期**：每次页面显示时静默调用 `refreshToolInfo()` 刷新 Coze 连接状态和 prompt_info
- **新增 `refreshToolInfo()` 方法**：单独请求 `/api/tools/:id`，仅更新 `cozeConnected`、`toolActivated`、`prompt_info`、`opening_statement`、`bot_available` 等字段，不重置消息列表，不显示 loading
- 使用 `this._toolId` 存储工具 ID，供 onShow 使用

### V30 补充修复 — 后端 Bot 信息拉取降级

#### 问题
即使 `cozeConnected` 为 true，`getValidCozeToken` 仍可能因为 token 即将过期（5分钟内）、refreshToken 刷新失败等原因抛异常，导致 Bot 信息（含 prompt_info）拉取失败，返回 null。

#### 修复
`src/app/api/tools/[id]/route.ts` 第 64-66 行：
- `getValidCozeToken` 失败时，降级使用 `getCozeToken` 直接获取 token（即使可能过期）
- 至少尝试拉取 Bot 信息，不因为 token 校验失败就放弃
- 添加 `console.warn` 日志记录降级原因

## V31 — Web端和小程序端快捷输入上传功能修复

### 问题
工具详情页"快捷输入"中图片/音频/视频的上传按钮无法正常使用。

### 根因分析
1. **Web端** (`src/lib/api-client.ts`)：`callAuthenticatedApi` 在处理上传请求时，通过展开运算符合并 headers，对于 FormData body 可能干扰浏览器自动设置的 `Content-Type: multipart/form-data; boundary=...`

2. **小程序端** (`miniprogram/pages/tool-detail/tool-detail.js`)：
   - `onMediaUpload` 中有未使用的死代码 `chooseFn` 变量
   - `wx.chooseMessageFile` 的 `type` 参数写死为 `'file'`，没有根据媒体类型区分 `'audio'` / `'video'`
   - 缺少 `extension` 参数限制可选文件类型

### 修改文件
1. **`src/lib/api-client.ts`** — 改进 headers 合并方式：
   - 改用 Object 遍历合并，避免展开运算符可能的行为不一致
   - 小写化 header key 便于比对
   - 对于 FormData body，不手动设置 Content-Type，让浏览器自动生成 boundary

2. **`miniprogram/pages/tool-detail/tool-detail.js`** — `onMediaUpload()`：
   - 移除死代码 `chooseFn` 变量
   - `wx.chooseMessageFile` 的 `type` 改为 `mediaType === 'audio' ? 'audio' : 'video'`
   - 新增 `extension` 参数限制可选文件类型（音频/视频各自支持的格式）

### V31 补充 — 上传时 "Internal S" 报错修复

#### 问题
上传文件时提示 `Unexpected token 'l',"Internal S".. is not valid JSON`。实际上是 Next.js 返回 500 HTML 错误页面（以 `<!DOCTYPE html>` 开头），前端用 `res.json()` 解析 HTML 导致报错。

#### 根因
1. **Web端** (`src/app/tools/[id]/page.tsx`)：`const data = await res.json()` 在 `res.ok` 检查之前调用，500 响应是 HTML 不是 JSON
2. **小程序端** (`miniprogram/pages/tool-detail/tool-detail.js`)：`JSON.parse(uploadRes.data)` 在 statusCode 非 2xx 时直接解析，data 可能是 HTML
3. **后端**：Route Handler 中某些未捕获异常可能导致 Next.js 返回 500 HTML 而不是 JSON

#### 修改文件
1. **`src/app/tools/[id]/page.tsx`** — 先检查 `res.ok`，不 ok 时 try-catch 解析 JSON 错误消息，解析失败则显示通用错误
2. **`miniprogram/pages/tool-detail/tool-detail.js`** — `onMediaUpload()` 中所有 `JSON.parse()` 调用加 try-catch，解析失败显示通用错误
3. **`src/app/api/upload/media/route.ts`** — POST handler 外层加 try-catch，确保任何未捕获异常都返回 JSON 格式错误
4. **`src/app/api/upload/image/route.ts`** — 同上
