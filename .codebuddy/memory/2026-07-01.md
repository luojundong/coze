## 修复小程序链接无法复制/打开的问题（情况A：纯文本显示）

### 根本原因
三个层面导致消息中的链接（如 TOS 视频链接）显示为纯文本，无法点击/长按：

1. **`processMessages`** (tool-detail.js:111) 对 `isStreaming: true` 的消息直接跳过 → 永远不生成 `parsed` 字段
2. **WXML 模板** (tool-detail.wxml:145) 使用 `wx:if="{{item.isStreaming}}"` 优先匹配流式消息，跳到纯文本渲染分支，完全跳过 `parsed` 的富文本渲染
3. **硬超时处理器** (tool-detail.js:512) 更新 content 时未设 `isStreaming: false`，消息永久卡在流式状态

### 修复内容

**tool-detail.js**:
- `enrichMessage()`: 添加 `_parsedContent` 缓存追踪，内容未变则跳过重复解析；移除 `isStreaming` 状态的依赖
- `processMessages()`: 只要有内容就调用 `enrichMessage`，不再检查 `isStreaming`
- 硬超时处理器 (line 512-518): 添加 `isStreaming: false`
- 轮询达到上限但有内容 (line 990-992): 添加 `isStreaming: false`

**tool-detail.wxml**:
- 助手消息渲染重构：`parsed` 存在时优先渲染富文本（无论是否流式），流式时在内容末尾显示闪烁光标
- 无 `parsed` 时回退到纯文本 + 加载状态

## 修复 Safari 打开 TOS 视频链接 AccessDenied 问题

### 根本原因
- TOS 签名 URL 默认有效期 `X-Tos-Expires=86400`（24 小时）
- 用户从 Coze 收到的链接虽是新签名的，但在 Safari 打开返回 `AccessDenied (EC: 0003-00000015, DetailErrCode: 14006)`
- 原因之一：Coze Bot/Workflow 的 message 内容可能不包含 TOS 完整 URL（仅含 markdown 引用），或签名实际已失效
- **关键问题**：`v3/chat/message/list` 返回的 `msg.content` 可能是**对象**（含 `file_id`/`file_url`），但原代码只处理字符串 → 找不到匹配 URL，刷新永远失败

### 修复内容

**src/app/api/workflow/refresh-url/route.ts**:
- 支持 `content` 为字符串 / 对象 / JSON 字符串三种格式
- 提取 `file_id`（如 `c.file_id`、`object_string.file_id`）
- 新增 `v1/files/{file_id}/content` API 调用作为备用签名刷新路径（Coze 会重定向到带新签名的 TOS URL）
- 多候选 URL 按"带签名 > 不带签名"优先级返回
