## 图片 404 问题最终解决 — Nginx 配置修复

**排查过程**：
1. 文件确认存在于 `/www/wwwroot/cozeZnt/public/uploads/image/`
2. `curl -sI` 显示响应 `x-powered-by: Next.js`，说明请求被 proxy_pass 到了 Node.js，而不是 Nginx 直接 serve
3. Nginx 配置中 `location ~ .*\.(gif|jpg|jpeg|png|bmp|swf)$` 是正则匹配，**优先级高于**前缀匹配的 `location /uploads/`
4. 图片 `.jpeg` 被正则 location 捕获，`root /www/wwwroot/cozeZnt` 拼接出 `/www/wwwroot/cozeZnt/uploads/image/xxx.jpeg`，但文件实际在 `/www/wwwroot/cozeZnt/public/uploads/image/`

**最终修复**（Nginx 配置）：
- 图片正则 location 的 `root` 从 `/www/wwwroot/cozeZnt` 改为 `/www/wwwroot/cozeZnt/public`
- 这样 Nginx 拼接路径为 `/www/wwwroot/cozeZnt/public/uploads/image/xxx.jpeg`，匹配实际文件位置
- 音频 `.mp3` 不受影响，因为 `.mp3` 不在正则匹配列表中，请求走 `location /` proxy_pass → Node.js server.ts 的静态文件拦截

**注意**：server.ts 中 `process.cwd()` → `resolve(__dirname, '..')` 的修复仍保留在 v30 中，作为兜底方案。但图片问题实际是 Nginx 层面解决的。

## 小程序端 — SSE 流式请求 120 秒超时导致"未收到回复"

**问题**：小程序显示"未收到回复"或"智能体超时"，Web 端正常。

**根因**：微信 `wx.request` 硬限制最大 `timeout: 120000ms`（120 秒），超过后强制断连。之前 SSE 流式请求设了 `timeout: 180000`（无效，微信会截断为 120s），且降级轮询逻辑依赖 SSE 成功接收到 task_id，但小程序 SSE 数据接收不稳定。

**最终方案（第三次迭代）**：**SSE 流式 + 独立兜底轮询 双模式并行**
- SSE 流式用于快速打字机效果
- 独立轮询通过 `task/run` 创建独立任务，与 SSE 同时运行
- SSE 成功 → 清理轮询并显示结果
- SSE 失败 → 独立轮询已在后台运行，无缝接管

**具体修改**（`miniprogram/pages/tool-detail/tool-detail.js`）：
1. 新增 `startDualMode()`：同时启动 SSE 和独立轮询
2. SSE 不再自己触发降级（noDataTimer 只用于日志）
3. SSE success/fail 回调简化：成功时清理轮询，失败时不处理（轮询已接管）
4. `hasReceivedChunk` 只在收到真正的文本内容时设置（不含 task_id 事件）
5. 独立轮询超时：240 秒（接近后端 300s maxDuration）
6. 轮询 completed/failed 状态检查：避免覆盖 SSE 已显示的内容
