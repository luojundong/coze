## 修复：小程序工具详情页 SSE 完成后仍然 stuck running / 不显示复制按钮

- 用户反馈：视频/图片链接类工具、纯文本回复类工具，内容已完整显示但底部仍显示「发送中...」和「停止」按钮，必须手动点停止才能再次发送，且复制按钮不出现。
- 同时反馈 G014 反推视频提示词工具出现超时错误（扣子后台已扣积分），内容已显示但同样卡 running。

### 根因分析

**根因 1: keep-alive / ping 包重置了无数据计时器**
- `noDataTimer` 之前只检测「最后一次收到任何 chunk」的时间 (`lastChunkTime`)
- 某些网络环境或 Coze SSE 会发送 keep-alive 空包/注释行，导致 `lastChunkTime` 持续刷新
- 内容其实已经停滞，但计时器无法触发自动释放

**根因 2: 后端 Bot SSE 流正常结束时只 `controller.close()`，没有显式发送 `done` 事件**
- 小程序前端依赖 `success` 回调来释放 running
- 在某些情况下 `success` 回调不可靠或延迟，导致 running 未释放

**根因 3: G014 超时错误是 Coze 智能体内部工具调用超时**
- 超时文案由 Coze 智能体返回，平台无法直接干预
- 扣积分发生在调用 Coze 之前（先扣费再调用），因此即使智能体返回超时/失败，积分也已扣除

### 具体修改

1. `miniprogram/pages/tool-detail/tool-detail.js`:

   - 新增 `lastContentUpdateTime` 变量，仅在实际收到/追加内容时更新
   - `noDataTimer` 增加「内容停滞检测」：当内容超过 15 秒未变化时，强制释放 running（不受 keep-alive 影响）
   - 在 `success` 和 `fail` 回调末尾增加兜底：如果 running 仍为 true，强制设为 false
   - 优化 `onShow`：如果最后一个 assistant 消息已有内容但仍在流式状态，强制释放 running + isStreaming

2. `src/app/api/workflow/stream/route.ts`:

   - Bot SSE 流正常结束时，在 `controller.close()` 前显式发送 `event: done\ndata: [DONE]`
   - 帮助小程序 `onChunkReceived` 在 `success` 不可靠时也能识别结束

## 修复：Web 端 Bot 工具图片上传后提示「下载图片链接超时」

- 用户反馈：Web 端上传图片后调用工具，Coze 返回错误「下载图片链接 https://coze.mooibi.com/uploads/... 超时了」，但图片在浏览器能正常访问。
- 根因：图片上传到本服务器后，Coze 服务端仍需从公网下载该 URL。本服务器到 Coze 的网络链路不稳定或 Coze 下载超时阈值较低，导致频繁超时。

### 具体修改

1. 新增 `src/lib/image-inline.ts`：

   - 从文本中提取本地上传的图片 URL（域名匹配 `PUBLIC_BASE_URL` 或路径 `/uploads/`）
   - 读取 `public/uploads/` 下对应文件，校验路径不能越界
   - 将图片转为 `data:image/xxx;base64,...` 数据 URI
   - 单张图片超过 4MB 时跳过内联，避免请求体过大
   - 提供 `inlineLocalImages`（文本）和 `inlineLocalImagesInObject`（递归对象）两个函数

2. 在三个 Bot 调用入口的 `additional_messages` 中应用：

   - `src/app/api/workflow/stream/route.ts`
   - `src/app/api/workflow/run/route.ts`
   - `src/app/api/workflow/task/run/route.ts`

   处理逻辑：
   - 先把 `parameters.input` 中的本地图片 URL 替换为 base64 data URI（保留在 text 消息中）
   - 再为每张图片追加一条 `content_type: 'image'`、`content: dataUri` 的多模态消息
   - Coze 无需再发起外部 HTTP 下载，直接解析 base64 图片，从根本上避免「下载图片链接超时」

## 修复：v45/v47 图片作为 base64 文本导致 Bot 回复“对不起，我无法回答这个问题”

- 用户反馈：Web 端上传图片和音频后，Bot 返回“对不起，我无法回答这个问题”。
- 根因：v47 移除 base64 独立图片消息后，图片 URL 仍被 `inlineLocalImages` 转成 base64 data URI 塞进文本消息。Bot 只能看到一长串 base64 字符串，无法作为图片处理，导致拒绝回复。
- 解决：把图片 URL 从文本中提取出来，单独作为 `content_type: 'image'` 消息发送；文本中只保留其它参数（如背景音乐、视频比例、key）。

### 修改文件
1. `src/lib/image-inline.ts` — 新增 `extractImageUrls(text, publicBaseUrl)` 和 `removeImageUrls(text)` 辅助函数
2. `src/app/api/workflow/stream/route.ts` — 提取图片 URL，追加为 image 消息
3. `src/app/api/workflow/run/route.ts` — 同上
4. `src/app/api/workflow/task/run/route.ts` — 同上，并把 imageUrls 传入 `executeCozeTask`

### 注意事项
- 图片 URL 必须是 Coze 可访问的公网 URL，否则 Coze 可能返回“下载图片链接超时”
- 音频 URL 仍作为文本发送，因为 Coze v3/chat 的 `content_type: 'image'` 只支持图片，不支持音频

## 修复：Web 端 workflow 工具报「错误: Request parameter error」

- 用户反馈：Web 端使用带有快捷输入参数表单的 workflow 工具（如 AI 视频生成）时，返回「错误: Request parameter error」。
- 根因分析：
  - 工具类型为 workflow，Coze 期望接收结构化参数（如 `image`、`mihe_key`、`camera_move`、`music`、`aspect_ratio`）。
  - 但 `src/app/tools/[id]/page.tsx` 的快捷输入表单点击「点击发送」后，无论 Bot 还是 Workflow 类型，都把参数拼接成 `input: "图片：xxx\n米核key：xxx..."` 这种自然语言文本，再通过 `/api/workflow/stream` 发给 Coze。
  - workflow 类型后端把 `{ input: "..." }` 整体作为 `parameters` 传给 Coze `/v1/workflow/run`，参数名不匹配，Coze 返回 Request parameter error。

### 修复内容

1. `src/app/tools/[id]/page.tsx`：
   - `handleRunWorkflow` 改为支持接收 `Record<string, string>` 结构化参数对象，优先使用该对象，否则回退到普通输入框的 `input`。
   - 快捷输入参数表单的「点击发送」按钮区分工具类型：
     - Bot 类型：保持原逻辑，把参数拼接成自然语言 `input` 调用 `handleSendChat`。
     - Workflow 类型：直接以 `param.name` 为键组装结构化参数对象，调用 `handleRunWorkflow(params)`。
   - 「执行工作流」按钮的 `onClick` 改为 `() => handleRunWorkflow()`，避免 React 把 MouseEvent 作为参数传入导致类型错误。
