## 小程序端：修复工具使用一直转圈圈问题
### 问题诊断
用户使用工具时流式消息一直显示转圈圈动画（`isStreaming: true` 不解除），可能原因：
1. Next.js 生产环境 `ReadableStream` / `TransformStream` SSE 响应可能不实时推送数据到客户端
2. `wx.request` 的 `onChunkReceived` 未触发 → `noDataTimer` 10s 后降级轮询
3. 降级轮询 `startFallbackPoll` 创建新任务，但如果网络不稳定或服务端响应慢，也可能卡住
4. **最关键**：没有任何绝对硬超时机制，如果 SSE + 轮询都失败，spinner 永远转圈

### 修复内容（`miniprogram/pages/tool-detail/tool-detail.js`）
- **新增 SSE 硬超时（90秒）**：`hardTimeoutTimer` — 90秒后强制设置 `isStreaming: false` + `running: false`，保留已有内容
- **新增降级轮询硬超时（60秒）**：`fallbackHardTimeout` — 轮询超过60秒强制结束
- **防止重复降级**：新增 `fallbackStarted` 标记，避免 `noDataTimer` 和 `fail` 回调同时触发降级
- **所有退出路径均清理定时器**：`success`、`fail`、`onChunkReceived` 错误事件、轮询完成/失败/超时都 `clearTimeout(hardTimeoutTimer)` / `clearTimeout(fallbackHardTimeout)`
- `startFallbackPoll` 新增 `hardTimeoutTimer` 参数，降级时清除 SSE 硬超时，启用轮询自身硬超时

## 小程序端：第二轮修复 — 降级轮询复用服务端 taskId
### 问题根因
降级轮询 `startFallbackPoll` 调用 `/api/workflow/task/run` 创建**新任务**（重复扣积分 + 重复执行），而非复用原 stream 端点后台 `taskPromise` 已创建的任务。小程序端无法获取服务端 taskId。

### 修复内容
**服务端 `src/app/api/workflow/stream/route.ts`**：
- SSE 响应开头注入 `event: task_id` + `data: {"task_id": "..."}` 事件（在透传第一个 Coze chunk 之前）
- 使用 `TextEncoder` 写入 SSE 格式事件行

**小程序 `miniprogram/pages/tool-detail/tool-detail.js`**：
- `startStreamMode` 新增 `serverTaskId` 局部变量
- `onChunkReceived` 中捕获 `task_id` 事件，保存 `serverTaskId`
- `startFallbackPoll` 新增 `serverTaskId` 参数
  - 有 `serverTaskId` → 直接轮询 `/api/workflow/task/status?taskId=xxx`（不创建新任务）
  - 无 `serverTaskId` → 兜底走旧的 `task/run` 创建新任务路径
- 两处降级触发点（noDataTimer、fail 回调）均传入 `serverTaskId`

## 小程序端：修复工具详情页"加载中..."卡死
### 问题现象
打开工具详情页一直显示"加载中..."转圈圈，无法进入工具使用页面。

### 根因分析
`/api/tools/[id]` 接口在加载 bot 类型工具时，会调用 Coze `/v1/bots/${coze_id}` API 获取开场白等信息。该请求**无超时控制**，Coze API 响应慢或不可达时整个请求挂起。小程序端 `app.request` 默认超时 120 秒，导致页面长时间卡住。

### 修复内容
**服务端 `src/app/api/tools/[id]/route.ts`**：
- 给 `fetch('/v1/bots/')` 加 AbortController + 8 秒超时
- 超时/错误后 catch 不再阻塞，返回基础信息（没有开场白等），页面正常加载

**小程序 `miniprogram/pages/tool-detail/tool-detail.js`**：
- `loadTool` 请求加 `timeout: 20000`，最多等 20 秒
