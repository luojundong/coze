## v19 部署：Coze 图像流并发限制自动重试 + 友好提示

### 问题
用户调用智能体时遇到 Coze 平台错误："当前同时执行图像流同一节点数量超过功能限制，请修改相关节点后重试"。
这是 Coze 官方的单用户并发限制（同一图像流节点最大并发 4 个），在平台内测试没问题，但 API 调用时容易触发。

### 修改内容

**服务端 - workflow/run/route.ts：**
- 添加 `isImageFlowConcurrencyError()` 函数识别错误（错误码 720712001 + 关键词匹配）
- 添加 `fetchWithRetry()` 函数：遇到图像流并发错误时等待 4-6 秒（随机 jitter 错峰）后重试，最多 2 次
- 错误响应增加 `retryable: true` 和 503 状态码
- 友好提示改为："当前使用人数较多，智能体中的图像处理节点排队拥挤，请稍后重试"

**服务端 - workflow/stream/route.ts：**
- 同步添加 `isImageFlowConcurrencyError()` 和 `fetchWithRetry()` 函数
- 主路径 SSE 请求、后台 taskPromise、非 SSE JSON 响应三处均增加了重试和友好提示

**Web 前端 - tools/[id]/page.tsx：**
- `handleRunWorkflow` 中识别 `retryable` 字段，友好展示而不抛异常
- SSE JSON 错误响应中识别 `retryable`，友好展示
- 降级轮询错误分类增加 "图像处理节点排队拥挤" 关键词

**小程序端 - miniprogram/tool-detail/tool-detail.js：**
- SSE JSON 错误响应中识别 `retryable`，友好展示
- 降级轮询错误分类增加 "图像处理节点排队拥挤" 关键词

## v20 部署：修复 stream/route.ts 后台 taskPromise 重复调用 Coze API

### 问题
`workflow/stream/route.ts` 中，SSE 主路径和后台 `taskPromise` 各自独立发起了一次 `/v3/chat` 请求。
这导致：
1. 同一个用户消息被 Coze 处理两次（消耗双倍 token/配额）
2. 自动重试时双方各自重试，进一步放大重复调用

### 修改内容
- 移除了后台 `taskPromise` 中独立的 Coze 请求
- 改为：SSE 主路径使用 `fetchWithRetry` 发起唯一一次 Coze 请求
- 通过 `cozeResponse.clone()` 将响应体复制给后台 taskPromise 使用
- 历史截断只需执行一次，两个路径共用

## v20 补充修复：fetchWithRetry 提前消费 SSE body 导致 clone 失败

### 问题
用户输入带链接的长文本（如小红书链接），智能体返回时前端报错：`Response.clone: Body has already been consumed.`

### 根因
`fetchWithRetry` 在第66行对每次请求都调用 `response.text()` 来检查错误码。对于 SSE 流式响应（content-type: text/event-stream），`response.text()` 会消费整个 body stream，之后 `cozeResponse.clone()` 就失败了。

### 修复（第一版 - 仍有问题）
先 clone 再读 text，确保原始 response 的 body 不被消费。但 `response.clone()` 在某些 Node.js fetch 实现中（body 被锁定）仍会失败。

### 修复（第二版 - 最终方案）
彻底避免 clone：对于 SSE 流式响应（content-type 包含 `text/event-stream`），**不提前读 body 做错误检查**，只检查 `response.ok` 就返回原始 response。body 完整保留给后续 SSE 透传和 clone 使用。非 SSE 响应才读 text 做 JSON 错误码检查。

## v21 部署：彻底移除 clone()，改为单流管道

### 问题
v20 的 clone() 方式在 Node.js undici fetch 中仍然不稳定，特定场景下 clone 会失败。

### 修改内容
- 完全移除 `cozeResponse.clone()` 和后台 taskPromise
- 改为单一 `ReadableStream`：从 `cozeResponse.body` 获取一个 reader
- 每读一块数据同时做两件事：`controller.enqueue()` 透传给前端 + 解析 SSE 事件更新 taskStore
- 不再需要双 reader 和 clone

## v22 部署：修复 fetchWithRetry 消费 body 后主函数再次读取的问题

### 问题
v21 部署后错误变为 `Body is unusable: Body has already been read`。
根因：`fetchWithRetry` 对非 SSE JSON 响应调用 `response.text()` 消费了 body，
但返回的 response 对象仍被传给主函数。主函数又尝试 `cozeResponse.text()` 或 `getReader()`，导致报错。

同时，工具详情页上传音频时，苹果/安卓手机均报类似错误。

### 修改内容
1. `fetchWithRetry` 返回值新增 `bodyConsumed: boolean` 标志位
   - SSE 200 OK → bodyConsumed=false（body 完整保留）
   - 其他所有路径（非 SSE / 错误响应）→ bodyConsumed=true
2. JSON 响应路径：`bodyConsumed` 时直接用 `cozeText`，不再调 `cozeResponse.text()`
3. SSE 路径增加 `bodyConsumed` 兜底检查，避免尝试 `getReader()` 已消费的 body

### 部署文件
- `coze-deploy-v22.zip` (约 7.7MB)
