# 2026-06-29 工作记录

## 小程序对话框滚动问题修复（修正版）
- **真正的问题**：部分手机型号上，`chat-area` 的 `scroll-view` 无法正常滚动，聊天区域（提示词/开场白）像是被固定住，看不到下方对话内容。用户反馈是"聊天区域被工具提示词固定住导致看不到内容"。
- **根因**：
  1. 部分安卓机型 `flex: 1` + `min-height: 0` 在 `scroll-view` 上不生效，高度计算为 0，导致无法滚动
  2. `scrollTop: 99999` 在部分机型上不可靠
  3. 缺少 `-webkit-overflow-scrolling: touch` 影响 iOS 滚动体验
- **修复方案**：
  1. **chat-area CSS**：添加 `height: 0`（配合 `flex: 1` 在 flex 布局中正确计算高度）、`box-sizing: border-box`、`-webkit-overflow-scrolling: touch`
  2. **message-list**：添加 `min-height: 100%` 确保内容能撑开 scroll-view
  3. **wxml**：`scroll-view` 添加 `scroll-into-view` 和 `scroll-with-animation` 属性；给每条消息添加 `id="msg-{{item.id}}"` 锚点
  4. **JS `_setData`**：双保险滚动策略 — 同时设置 `scrollTop: 99999` 和 `scrollIntoViewId` 指向最后一条消息
  5. **param-form**：恢复为独立的 `scroll-view` 包裹（`param-form-scroll`），保持原有独立滚动行为，`max-height: 360rpx`
- **修改文件**：
  - `tool-detail.wxml`：chat-area 添加 `scroll-into-view`/`scroll-with-animation`；消息添加 `id`；param-form 恢复外层 scroll-view 包裹
  - `tool-detail.wxss`：chat-area 添加 `height: 0`/`box-sizing`/`-webkit-overflow-scrolling`；message-list 添加 `min-height: 100%`；新增 `.param-form-scroll` 样式
  - `tool-detail.js`：添加 `scrollIntoViewId` 数据字段；`_setData` 中同时使用 scrollTop 和 scroll-into-view 双保险

## 小程序消息中链接无法点击/复制问题修复
- **问题**：智能体返回的消息中包含链接（如 https://...），用户无法点击打开，也无法复制
- **根因**：链接用 `<text>` 元素 + `bindtap` 渲染，在小程序 `scroll-view` 内部 `<text>` 的 `bindtap` 事件经常不触发（尤其在某些安卓机型）
- **修复方案**：
  1. 将链接渲染从 `<text>` 改为 `<view>` 包裹，增强点击热区
  2. 链接样式改为卡片式（浅蓝背景 + 边框），提高可辨识度和可点击性
  3. 点击后复制链接到剪贴板并弹窗提示
- **修改文件**：
  - `tool-detail.wxml`：系统消息和助手消息中的链接元素改为 `<view class="msg-link">` + `<text class="msg-link-text">` 结构
  - `tool-detail.wxss`：`.msg-link` 改为 `display: block` + 卡片式样式（`background: #EFF6FF`、`border: 1rpx solid #BFDBFE`、`border-radius: 8rpx`、`padding: 12rpx 16rpx`）；新增 `:active` 态

## 小程序链接无法打开/复制问题 — 后续修复（6/29）
- **用户反馈**：链接复制到浏览器后提示 `InvalidAccessKeyId`，说明链接本身的 AccessKey 已失效；小程序内点击链接没有反应
- **用户补充**：web 端同一个智能体的链接可以正常打开播放，小程序端不行 → 问题不在 Coze 端 AK，在小程序前端解析逻辑
- **根因**：`parseContent` 中的 `isImageUrl` 函数用正则 `/\/\/.*\/tos-.*\//` 匹配 TOS 存储桶 URL，导致 `.mp4` 视频链接被错误判定为 `type: 'image'`，渲染为 `<image>` 组件（预览图片）而非可点击的链接卡片
- **前端修复**：
  1. `isImageUrl` 增加 `nonImageExtRe`（视频/音频/文档扩展名）检测：如果 URL 以 `.mp4`/`.avi` 等结尾，优先排除，不判为图片
  2. 为两处 `msg-link` 元素增加 `bindlongpress="openLink"` 长按复制支持
  3. `openLink` 增加 `fail` 回调和视频链接检测
